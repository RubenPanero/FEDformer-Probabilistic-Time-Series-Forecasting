# Plan: Optimize Critical Bottlenecks — FEDformer Performance Suite

## Branch Name
`optimize-critical-bottlenecks`

## 1. Executive Summary

This plan implements **Phase 1 (Critical Bottlenecks)** from the optimization docs — the HIGH IMPACT, LOW EFFORT optimizations identified through profiling analysis. The goal is a **30-40% reduction in training time** with **<1% accuracy impact** on canonical metrics (Sharpe, Sortino, MaxDD).

## 2. Optimizations to Implement

### OPT-1: Configurable MC Dropout Samples (Impact: -40% eval time)

**Description:** Replace hardcoded `n_samples=50` in `_evaluate_model()` with a configurable value. Default to 20 for training (sufficient for quantile estimation), keep 50 for inference.

**Locations:**
- `config.py` — Add `mc_samples_train` and `mc_samples_inference` to `LoopSettings`
- `training/trainer.py:696-697` — Change hardcoded `n_samples=50` to use config
- `training/utils.py:18-22` — Default `n_samples=100` → use caller-provided value (already does, just need caller to pass correct value)
- `main.py` — Add `--mc-samples-train` and `--mc-samples-inference` CLI flags

**Changes Required:**
1. Add to `LoopSettings`:
   ```python
   mc_samples_train: int = 20
   mc_samples_inference: int = 50
   ```
2. In `training/trainer.py:_evaluate_model()`, line 696-697:
   ```python
   # Current:
   samples = mc_dropout_inference(model, batch, n_samples=50, use_flow_sampling=True)
   # New:
   samples = mc_dropout_inference(
       model, batch, n_samples=self.config.mc_samples_train, use_flow_sampling=True
   )
   ```
3. Add CLI flags in `main.py` argparse

**Dependencies:** None — pure config addition, no structural changes.

**Risk:** Very low — research shows 20 samples sufficient for p10/p50/p90 quantile estimation.

---

### OPT-2: Persistent DataLoader Workers + Prefetch (Impact: -10-15% data loading)

**Description:** Enable `persistent_workers=True` and `prefetch_factor=2` on the train DataLoader. Currently `persistent_workers=False` is explicitly set for "safe memory policy between folds", but this is overly conservative — we can enable persistence within a fold and shut down between folds.

**Locations:**
- `training/trainer.py:234-245` — `_prepare_data_loaders()` train_loader creation
- `training/trainer.py:266-280` — `_make_loader()` creation

**Changes Required:**
1. In `_prepare_data_loaders()`, for `train_loader` only:
   ```python
   # Current:
   persistent_workers=False,
   # New:
   persistent_workers=num_workers > 0,  # Enable when workers exist
   prefetch_factor=2 if num_workers > 0 else None,
   ```
2. Keep `persistent_workers=False` for `test_loader` (evaluation is one-shot per fold)
3. The existing per-fold GC cleanup (`del components` + `torch.cuda.empty_cache()`) already handles memory between folds

**Dependencies:** None — the existing fold-level cleanup already prevents memory leaks.

**Risk:** Low — `persistent_workers` only keeps workers alive within a single fold's training loop. Workers are recreated per fold since DataLoaders are rebuilt each fold.

### OPT-3: Fused Affine Transform in Coupling Layers (Impact: -10% flow forward)

**Description:** Replace the element-wise `x2 * s.exp() + t` with `torch.addcmul(t, x2, scale)` to reduce memory allocations and kernel launches in `AffineCouplingLayer.forward()`.

**Locations:**
- `models/flows.py:35-54` — `AffineCouplingLayer.forward()`
- `models/flows.py:56-73` — `AffineCouplingLayer.inverse()` (same pattern)

**Changes Required:**
1. In `forward()`:
   ```python
   # Current:
   s, t = params[0], params[1]
   s = torch.tanh(s)
   y2 = x2 * s.exp() + t
   # New:
   s, t = params[0], params[1]
   scale = torch.exp(torch.tanh(s))
   y2 = torch.addcmul(t, x2, scale)  # t + x2 * scale
   log_det_jacobian = torch.log(scale).sum(dim=-1)
   ```
2. In `inverse()`:
   ```python
   # Current:
   s, t = params[0], params[1]
   s = torch.tanh(s)
   x2 = (y2 - t) * (-s).exp()
   # New:
   s, t = params[0], params[1]
   scale = torch.exp(torch.tanh(s))
   x2 = (y2 - t) / scale  # Division is faster than exp+neg+mul
   ```

**Dependencies:** None — pure mathematical equivalence.

**Risk:** Very low — `addcmul` is a standard fused operation. `log_det_jacobian` must use `torch.log(scale)` instead of `s.sum()` since scale = exp(tanh(s)), so log(scale) = tanh(s).

### OPT-4: Batched MC Dropout Inference (Impact: -15% MC overhead)

**Description:** Process MC samples in micro-batches to reduce Python interpreter overhead and memory fragmentation. Instead of 50 individual forward passes, process in groups of 10.

**Locations:**
- `training/utils.py:18-95` — `mc_dropout_inference()`

**Changes Required:**
1. Add `batch_size` parameter to `mc_dropout_inference`:
   ```python
   def mc_dropout_inference(
       model: nn.Module,
       batch: dict[str, torch.Tensor],
       n_samples: int = 100,
       use_flow_sampling: bool = True,
       mc_batch_size: int = 10,
   ) -> torch.Tensor:
   ```
2. Pre-allocate output tensor and process in micro-batches:
   ```python
   # Pre-allocate
   first_sample = _get_single_sample(...)
   sample_shape = first_sample.shape
   all_samples = torch.empty(n_samples, *sample_shape, device=device)
   all_samples[0] = first_sample
   
   # Process remaining in batches
   for batch_start in range(1, n_samples, mc_batch_size):
       batch_end = min(batch_start + mc_batch_size, n_samples)
       for i in range(batch_start, batch_end):
           dist = model(x_enc, x_dec, x_regime)
           all_samples[i] = dist.sample(1)[0]
   ```

**Dependencies:** OPT-1 (configurable n_samples) — batched processing is most beneficial when n_samples is known upfront.

**Risk:** Low — same mathematical result, just reorganized loop.

### OPT-5: Configurable FFT Modes (Impact: -10% forward pass)

**Description:** Make the default `modes` configurable at 48 instead of 64, since modes beyond 48 capture high-frequency noise with diminishing returns.

**Locations:**
- `config.py:41` — `TransformerSettings.modes` default
- `models/layers.py:78-80` — `FourierAttention.__init__` (already clamps modes)

**Changes Required:**
1. No code change needed — `modes` is already configurable via `TransformerSettings`. The optimization plan suggests testing `modes=48` vs `modes=64`.
2. Add documentation/CLI flag `--modes` if not already present
3. The clamping logic at `models/layers.py:80` already handles edge cases: `self.modes = min(modes, max(1, seq_len // 2))`

**Dependencies:** None — already implemented, just needs validation testing.

**Risk:** Minimal — modes 48-64 capture noise; 25% fewer modes = 25% faster FFT.
